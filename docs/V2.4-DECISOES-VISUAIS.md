# ServiceDesk Toolkit Corporate - Decisões Visuais v2.4

## 1. Decisão principal

A v2.4 não será focada em adicionar novas funções.

A prioridade da v2.4 será transformar o Toolkit em uma ferramenta corporativa guiada, visualmente mais limpa e operacionalmente mais segura.

## 2. Sidebar

A sidebar não deve ser uma lista completa de ferramentas.

A sidebar deve conter apenas entradas principais e atalhos de alto valor.

### Fica na sidebar

- Início
- Atendimento Rápido
- Ferramentas Técnicas
- Evidências
- Administração do Toolkit

### Sai da sidebar

- Botões técnicos específicos demais
- Ações duplicadas que já existem dentro das abas
- Correções críticas
- Ferramentas avançadas de rede
- Ferramentas avançadas de Windows
- Ações de reset/reparo pesado

## 3. Home

A Home deve virar uma tela executiva e operacional.

### Home deve mostrar

- Nome do Toolkit
- Versão atual
- Canal atual
- Hostname
- Usuário
- Status de administrador
- Status geral da máquina
- Botão para Atendimento Rápido
- Botão para gerar pacote de suporte
- Botão para abrir evidências
- Botão para Base de Conhecimento

### Home não deve mostrar

- Botões técnicos demais
- Ações críticas
- Funções duplicadas da sidebar
- Informações longas que pertencem ao painel Resultado

## 4. Atendimento Rápido

O Atendimento Rápido deve ser o fluxo principal para N1.

### Fluxos oficiais

- Sem internet
- VPN / Appgate
- Teams / Outlook
- Impressora
- Windows Update
- Máquina lenta

### Saída padrão de cada fluxo

- Diagnóstico
- Causa provável
- Próxima ação
- Evidência para chamado

## 5. Ferramentas Técnicas

Ferramentas Técnicas devem ser separadas por categoria.

### Categorias

- Rede
- Windows / Reparo
- Teams / Office
- Store / Apps
- Impressoras
- Segurança
- GPO / Sistema
- Teste TCP

## 6. Evidências

A área Evidências deve centralizar tudo que ajuda a fechar ou escalar chamado.

### Deve conter

- Inventário completo
- Diagnóstico de rede
- Relatório HTML
- Relatório TXT
- Pacote de suporte
- Abrir pasta de relatórios
- Copiar resultado

## 7. Administração do Toolkit

A administração deve cuidar da própria ferramenta.

### Deve conter

- Status do Toolkit
- Atualizar Toolkit
- Rollback dry-run
- Validar instalação
- Abrir logs
- Abrir backups
- Último resumo de update
- Resumo de logs

## 8. Classificação dos botões

Todos os botões devem seguir uma classificação visual.

### Diagnóstico

Baixo risco. Apenas coleta informação.

Exemplos:

- Inventário
- Status IP
- Ver DNS
- Ver rotas
- Status Appgate
- Status Teams / Office

### Correção segura

Médio risco. Altera algo simples ou reversível.

Exemplos:

- Flush DNS
- Renovar IP
- Sincronizar horário
- Reiniciar spooler
- Limpar temporários

### Ação crítica

Alto risco. Pode alterar rede, sistema, cache sensível ou exigir reinício.

Exemplos:

- Reset Winsock
- Reset TCP/IP
- DISM
- SFC
- Limpar cache Windows Update
- Corrigir Appgate
- Corrigir TPM / Office
- Limpar fila de impressão

## 9. Regra para ações críticas

Toda ação crítica deve ter:

- Botão visualmente destacado
- Mensagem de risco
- Confirmação explícita
- Log de ação
- Texto informando impacto possível

## 10. Decisão de UX

O técnico deve sempre conseguir responder rapidamente:

- O que eu clico primeiro?
- Isso só diagnostica ou altera a máquina?
- Qual evidência eu posso anexar no chamado?
- Qual é a próxima ação recomendada?
- Quando devo escalar para N2/N3?

## 11. Decisão de release

A v2.4.0 só deve virar stable quando:

- Sidebar estiver reorganizada
- Home estiver mais clara
- Atendimento Rápido estiver padronizado
- Ações críticas estiverem protegidas
- Quality Gate passar
- Release Validation passar
- Installed Validation passar
- Documentação estiver atualizada
