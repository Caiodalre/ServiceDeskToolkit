# ServiceDesk Toolkit Corporate - Plano Visual e Funcional v2.4

## 1. Objetivo da v2.4

Transformar o ServiceDesk Toolkit Corporate em uma central corporativa guiada para atendimento técnico, com foco em triagem, coleta de evidências, correções seguras e padronização operacional.

A v2.4 não deve priorizar quantidade de botões. Deve priorizar clareza, fluxo, segurança e efetividade no atendimento.

## 3. Problema identificado

O app está funcional, mas começa a correr risco de virar uma central de botões.

Principais riscos:

- Sidebar com ações demais
- Duplicidade entre Home, Sidebar, Abas e Atendimento Rápido
- Falta de hierarquia visual clara
- Usuário técnico pode não saber o que clicar primeiro
- Ações de diagnóstico e ações corretivas aparecem próximas demais
- Ferramentas avançadas ainda não estão suficientemente separadas
- Relatório e evidências precisam ter papel claro
- O app precisa parecer uma ferramenta corporativa, não um script acumulado

## 4. Princípio de produto

A v2.4 deve seguir este equilíbrio:

- 70% diagnóstico e coleta de evidência
- 20% correção segura guiada
- 10% ação administrativa avançada

O Toolkit não deve prometer resolver tudo automaticamente. Ele deve acelerar triagem, padronizar evidências e reduzir erro humano.

## 5. Público-alvo

Usuários principais:

- Service Desk N1
- Analista de suporte
- Analista de infraestrutura
- Técnico que precisa coletar evidências antes de escalar
- Técnico iniciante que precisa de fluxo guiado

Usuários secundários:

- N2 / N3 para análise de evidências
- Liderança técnica para auditoria e padronização

## 6. Arquitetura visual proposta

A interface deve ser organizada em 5 áreas principais:

### 6.1 Início

Objetivo: dar visão rápida da máquina e orientar o próximo passo.

Deve conter:

- Status da máquina
- Status de administrador
- Hostname
- Usuário
- IP principal
- Windows
- Botões principais:
  - Abrir Atendimento Rápido
  - Gerar pacote de suporte
  - Gerar diagnóstico do Toolkit
  - Abrir Base de Conhecimento

### 6.2 Atendimento Rápido

Objetivo: resolver ou direcionar os chamados mais comuns.

Fluxos prioritários:

- Sem internet
- VPN / Appgate
- Teams / Outlook
- Impressora
- Windows Update
- Máquina lenta

Cada atendimento deve retornar:

- Diagnóstico
- Causa provável
- Próxima ação
- Evidência para chamado

### 6.3 Ferramentas Técnicas

Objetivo: reunir ferramentas para técnicos que já sabem qual ação executar.

Categorias:

- Rede
- Windows / Reparo
- Teams / Office
- Store / Apps
- Impressoras
- Segurança
- GPO / Sistema
- Teste TCP

### 6.4 Evidências

Objetivo: facilitar fechamento de chamado e escalonamento.

Deve conter:

- Inventário completo
- Diagnóstico de rede
- Relatório HTML
- Relatório TXT
- Pacote de suporte
- Abrir pasta de relatórios
- Copiar resultado

### 6.5 Administração do Toolkit

Objetivo: manutenção da própria ferramenta.

Deve conter:

- Status do Toolkit
- Atualizar Toolkit
- Rollback dry-run
- Validar instalação
- Abrir logs
- Abrir backups
- Último resumo de update
- Resumo de logs

## 7. Nova regra para Sidebar

A sidebar deve ser navegação e atalhos essenciais, não uma lista completa de funções.

Sidebar recomendada:

- Início
- Atendimento Rápido
- Ferramentas Técnicas
- Evidências
- Administração

Ações específicas devem ficar dentro das abas correspondentes.

## 8. Nova regra para botões

Cada botão deve ser classificado como:

### Diagnóstico

Apenas coleta informação. Baixo risco.

Exemplos:

- Inventário
- Status IP
- Ver DNS
- Ver rotas
- Status Appgate
- Status Teams / Office

### Correção segura

Altera algo simples ou reversível. Médio risco.

Exemplos:

- Flush DNS
- Sincronizar horário
- Reiniciar spooler
- Limpar temporários

### Ação crítica

Altera componentes sensíveis. Alto risco.

Exemplos:

- Reset Winsock
- Reset TCP/IP
- DISM
- SFC
- Limpar cache Windows Update
- Corrigir Appgate
- Corrigir TPM / Office
- Limpar fila de impressão

Ações críticas devem ter confirmação clara, mensagem de risco e log.

## 10. Critério de efetividade

A v2.4 será considerada efetiva se conseguir:

- Reduzir tempo de triagem inicial
- Padronizar coleta de evidências
- Ajudar técnico iniciante a saber o próximo passo
- Evitar execução de correções perigosas sem confirmação
- Facilitar escalonamento para N2/N3
- Tornar o atendimento mais documentado
- Reduzir comandos manuais repetitivos
